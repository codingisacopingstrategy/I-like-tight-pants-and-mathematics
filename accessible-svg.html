Accessible svg

Implementing SVG

I know that SVG's are supported in Firefox and Safari, so I want them on this website. I figured I could best look how it is done in the wild. I looked at the site of Doug Schepers because he works for the W3C SVG working group so he must know what he is doing. This is how he embeds an SVG graphic in his HTML:

<object type="image/svg+xml" width="420" height="120" data="http://schepers.cc/svg/path/pathpoints.svg">Please use a modern browser.</object>

Very simple. The object tag can be used to embed any kind of content. The type is specified by the type attribute and if the browser can not display this particular kind of content, it will fall back and display what's contained within the object tag. In this case the text: 'Please use a modern browser.'.

The message looks a bit rude though. I suppose it is okay for Doug because he is part of the working group and they are pushing their format hard, and they want people to take notice when browsers do not support the standards they are working so hard at. In this case, the browser which is not being cooperative is Internet Explorer (up until version 8).

I do not think I can afford making such a statement at the viewers expense though. Most people who visit my site probably use Firefox, Chrome or Safari. But people who work for larger cultural organisations, or for businesses or governments, will often have to put up with Internet explorer. Often they do not even want to use this browser, but the IT department mandates it. I would not like it if someone who has to decide over a funding application or an exhibition or a sponsorship is unable to access my content because he happens to be using a suboptimal computer.

Then there is also the possibility that people are using a web browser that does not display images. Such browsers exist, lynx is an example. More importantly, there are people who can not see, and therefore use programs called screen readers that read the web page to them. We have to take them into account too.

So this is how I did it, you will find the SVG on the page I like tight pants and the problem with web apps.

<p>
  <object type="image/svg+xml" style="width:736px;height:880px;" data="/and/assets/called/theproblemwithwebapps.svg">
    <img src="/and/assets/called/theproblemwithwebapps.png" style="width:736px;height:880px;" alt="'Apps become like this:' a circle labelled APP encapsulating a circle labelled DATA. The app interfaces with a square with rounded corners labeled USER, as shown by two way arrows. 'While potentially they could be:' the user interfaces with a circle APP which interfaces with a separate circle DATA. 'The latter could enable': the USER interfaces with an APP which interfaces with an APP which interfaces with an APP which interfaces with DATA. 'As well as:' the USER interfaces with three APPS that all interface with DATA. '(Or any combination of the two)' " />
  </object>
</p>

I saved the image both as a SVG picture and a traditional PNG picture. Most browsers display the SVG. Internet Explorer falls back to the image tag contained within the object tag, and shows the PNG. Text only browsers and screen readers fall back from the object tag to the image tag, and instead of the image they display the alt text (screenshot).

SVG in the wild

You will not find too many web sites using SVG directly. Wikipedia, which hosts a lot of SVG's, converts them to PNG before displaying. Other websites use a JavaScript library like Raphael to render vector graphics.

Part of this has to do with the lack of support from Internet Explorer, but when I used SVG on a private wiki a while back another reason became apparent: rendering SVG's can be slow and unstable. It managed to crash Safari a couple of times.

This makes sense: there is no reason for browser vendors to spend a lot and time of effort in optimising a feature that is not used much. However, as long as it functions lackluster, people will also be less prone to use the feature.

I do not care for Catch 22's, so I say bring on the raw SVG (Rip ODB). Maybe browsers handle it better than they did a year ago. Maybe they will not. But in that case, an increase in bug reports will show the vendors soon enough that this format is to be reckoned with, and they will improve their rendering.

Additional points:

One other potentially interesting way to use SVG is to directly put the SVG code inside the HTML. There are a whole lot of caveats for this with XHTML, but it works differently in HTML5. I'll have to look into this.

If you want to see some SVG pictures, there are a lot of them to be found at Wikimedia Commons. And big up to the Open Clipart Library, a group of volunteers that has already produced more than 20.000 images in the format, all available in the public domain.

What is SVG?

A very popular way to describe an image is as a matrix of lots of small squares. This is called a raster image. Rasterised images are convenient because a computer screen is a raster too. On the internet raster formats are predominant, JPEG and PNG. Normally, the squares in the picture map to the squares of the screen.

The other popular way to describe an image as a collection of curves and shapes, called vectors, that can be described by mathematical formulas. The shape of the curves is determined by control points. Shapes are created by filling curves with colour.

Displaying vectors on the raster of the computer screen is a bit like plotting formulas on a graphing calculator. This can be performed at any resolution you want.

You can see this in effect with computer typefaces, which are also stored in a vector format: you can display a typeface at 12pt and 72pt and at both sizes the curves will look smooth.

That vector graphics perform independent of display resolution is notable enough for a vector format to be named after it: SVG, or Scalable Vector Graphics. This is the file format that has been adopted as a vector format for the web.

(SVG has more going for it than just being scalable: it is a web-standard that ties in well with the other web technologies, HTML, CSS and JavaScript. For example, you can make parts of your drawing interactive with JavaScript. )

Why has SVG not made it big (yet)?

Vector graphics shine when you scale your page. All major browsers now implement full page zooming, meaning you can zoom in on the web page with ï£¿ + or Ctrl +. While raster based images will get fuzzy, vector based images retain a sharp outline. Likewise, when you print a webpage, vector images come off much cleaner than rasters, since you have scaled to a higher resolution device.

Yet in daily practice most people do not zoom in on websites, and webpages are not printed all that often. That is why vector graphics have not been sorely missed; scalability remains a theoretical advantage.

All this changes with the new generation of mobile devices. The user interface of the iPhone does not operate on a fixed resolution. Rather it loads the whole page into and subsequently allows you to do a continuous zoom onto relevant parts.

On the high resolution of displays of contemporary mobile phones you will be looking at razor sharp text and fuzzy images.
